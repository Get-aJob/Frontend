# 알림 기능 기획서 초안 (WebSocket)

> 상태: 초안 — 백엔드·프론트 합의 후 세부 스펙(필드명·엔드포인트·이벤트 타입)을 확정한다.

## 1. 목표

- 실시간으로 발생하는 알림을 **WebSocket**으로 수신한다.
- **알림 목록 페이지**에서 사용자가 볼 수 있는 알림만 조회·필터·읽음 처리한다.
- **로그인 직후** 로그인 이전에 쌓인 **미확인(미읽음) 알림**이 있으면 사용자에게 노출한다(배지, 토스트, 드로어 등 — UI는 별도 확정).

## 2. 용어

| 용어 | 설명 |
|------|------|
| 알림(Notification) | 특정 사용자에게 귀속되는 단일 이벤트 레코드(제목, 본문, 링크, 타입, 생성 시각 등). |
| 미확인 / 미읽음 | 사용자가 아직 “봤다”로 처리하지 않은 알림. 제품 정책에 따라 “배지만 사라짐”과 “읽음 처리”를 분리할 수 있음(아래 7절). |
| 실시간 수신 | WS로 서버가 클라이언트에 푸시하는 알림. |
| 히스토리 | DB에 저장된 과거 알림 — REST(또는 GraphQL)로 페이지네이션 조회. |

## 3. 사용자 시나리오 (요약)

1. 사용자 A가 로그인한다 → 클라이언트는 WS 연결 + 미읽음 개수(또는 목록 일부)를 조회한다 → 미읽음이 있으면 UI에 표시한다.
2. 다른 사용자 또는 시스템이 A에게 해당하는 이벤트가 발생한다 → 백엔드가 알림을 저장하고 A의 활성 WS 세션에 브로드캐스트한다.
3. A가 알림 페이지에 들어간다 → 목록을 스크롤하며 과거 알림을 본다 → 개별/일괄 읽음 처리한다.
4. 탭을 닫거나 네트워크가 끊기면 WS는 끊긴다 → 재연결 후 `since` 또는 `lastSeenId` 기반으로 누락분을 보정할지 정책 결정(선택).

## 4. 백엔드에서 할 일 (초안)

### 4.1 데이터 모델 (예시)

- `notifications` (또는 동등 테이블)
  - `id`, `user_id`(수신자), `type`, `title`, `body`, `payload`(JSON: 이동 URL, 관련 엔티티 id 등), `created_at`
  - `read_at` (nullable) 또는 별도 `notification_reads` — **읽음 시각** 저장
- (선택) `delivered_at` / `seen_at` — 푸시 도달 vs 목록 열람 구분 시

인덱스: `(user_id, created_at DESC)`, `(user_id, read_at IS NULL)` 등 조회 패턴에 맞게.

### 4.2 REST (또는 동등) API — 히스토리·읽음

| 역할 | 메서드·경로 (예) | 설명 |
|------|------------------|------|
| 목록 | `GET /notifications?cursor=&limit=&unreadOnly=` | 페이지네이션, 미읽음만 필터 옵션 |
| 미읽음 개수 | `GET /notifications/unread-count` | 로그인 직후 배지·헤더용 |
| 읽음 처리 | `PATCH /notifications/:id/read` | 단건 읽음 |
| 일괄 읽음 | `POST /notifications/read-all` | (정책에 따라) 전체 또는 특정 조건까지 읽음 |

응답에 `totalUnread` 또는 목록 메타를 넣어 프론트가 캐시와 동기화하기 쉽게 한다.

### 4.3 WebSocket

- **연결 시 인증**: 쿼리 스트링 토큰은 피하고, **첫 메시지로 토큰 전송** 또는 **HTTP 업그레이드 전 쿠키/세션** 등 팀 표준에 맞는 방식으로 인증(보안 검토 필수).
- **구독 단위**: 사용자별 room / `userId` 기준으로만 해당 클라이언트에 전달.
- **서버 → 클라이언트 이벤트 (예시 타입명)**  
  - `notification:new` — 본문은 REST와 동일 스키마에 가깝게(프론트가 목록에 merge 가능).  
  - (선택) `notification:read` — 다른 탭에서 읽음 처리 시 동기화.  
  - (선택) `system:pong` — 하트비트.
- **클라이언트 → 서버 (선택)**  
  - `ping`, `subscribe` 확인용 등 — 최소화 권장.

### 4.4 알림 생성 트리거

- 도메인 서비스(지원, 댓글, 공고 마감 등)에서 **트랜잭션 내 또는 직후** 알림 레코드 생성 + WS 발행.
- 발행 실패 시: DB에는 남고 WS만 실패 → 다음 로그인/목록 조회로 복구 가능하도록 **저장 우선** 원칙 권장.

### 4.5 권한·보안

- `user_id`가 요청자와 일치하는 알림만 조회·읽음 처리 가능.
- Rate limit: WS 재연결 폭주, REST 목록 남용 방지.
- `payload`에 민감정보 최소화; 딥링크는 권한 검증은 **이동 후 페이지/API**에서 재수행.

### 4.6 운영

- 연결 수, 메시지 큐 적체, DB 슬로우 쿼리 모니터링.
- (선택) Redis Pub/Sub 등으로 웹소켓 인스턴스 간 브로드캐스트.

## 5. 프론트엔드에서 할 일 (초안)

### 5.1 알림 페이지

- 라우트(예: `/notifications`) — 로그인 가드.
- 목록 UI: 무한 스크롤 또는 커서 페이지네이션, 타입별 아이콘/색, 클릭 시 상세 또는 `payload`의 URL로 이동.
- 읽음 처리: 항목 진입 시 / 버튼 클릭 시 — 백엔드 정책에 맞춤.
- 빈 상태·에러·재시도.

### 5.2 WebSocket 클라이언트

- **연결 시점**: 로그인 성공 후(액세스 토큰 확보 시), 단일 싱글톤 또는 React Context Provider로 관리.
- **재연결**: 지수 백오프 + 최대 재시도; 백그라운드 탭에서는 연결 유지 정책(배터리) 선택.
- **수신 시**: 전역 스토어(또는 React Query 캐시)에 알림 추가, `unreadCount` 증가.
- **연결 해제**: 로그아웃 시 명시적 close 및 상태 초기화.

### 5.3 로그인 직후 “확인 못한 알림” 표시

1. 로그인 완료 후 `GET /notifications/unread-count` (또는 `unreadOnly=true`로 소량 목록) 호출.
2. `count > 0`이면 헤더 배지, 토스트, 모달, 알림 드롭다운 등 **제품이 정한 UI**로 표시.
3. (선택) WS 연결 전에 REST로 먼저 받아 두고, 이후 `notification:new`로 실시간 보강.

### 5.4 상태 동기화

- 목록 페이지와 헤더 배지의 `unreadCount`가 어긋나지 않게: 읽음 API 성공 시 로컬 상태·캐시 동시 갱신.
- 여러 탭: `BroadcastChannel` 또는 서버 `notification:read` 이벤트로 선택 동기화.

### 5.5 비로그인 / 만료

- 비로그인 시 WS 미연결, 알림 라우트 리다이렉트.
- 토큰 만료 시 WS 끊김 → 리프레시 후 재연결 플로우 정의.

## 6. 프론트–백 계약 체크리스트 (합의용)

- [ ] 알림 JSON 스키마 필드 확정 (`id`, `type`, `title`, `body`, `payload`, `createdAt`, `readAt` …)
- [ ] WS 이벤트 이름·페이로드 = REST 단건과 동일 여부
- [ ] 읽음 정의: 목록 스크롤만으로 읽음인지, 클릭 시인지, 별도 버튼인지
- [ ] 페이지네이션: cursor vs offset
- [ ] 인증 방식: WS handshake, 첫 프레임, 쿠키 도메인
- [ ] 알림 타입 목록(ENUM) 및 각 타입별 딥링크 규칙

## 7. 결정이 필요한 정책 (초안 메모)

- **“확인”과 “읽음” 분리 여부**: 배지 제거만 할지, DB `read_at`을 언제 찍을지.
- **보관 기간**: 오래된 알림 자동 삭제/아카이브.
- **실시간 필수 여부**: WS 불가 환경에서 폴링 폴백할지.

## 8. 구현 단계 제안 (MVP → 확장)

1. **MVP**: REST 목록 + unread-count + 단건 읽음 + WS `notification:new`만.
2. **2단계**: 일괄 읽음, 타입 필터, 재연결 후 누락 보정.
3. **3단계**: 다탭 동기화, 푸시 알림(모바일/PWA) 연계 검토.
