대상 파일: `src/view/Notification.tsx`  
관련 기획: `docs/알림_기능_기획서_초안.md`

## 1. 현재 페이지가 담고 있는 기능

### 1.1 데이터 모델 (목업)

| 필드 | 설명 |
|------|------|
| `id` | 알림 고유 식별자 |
| `type` | `COMMENT` · `APPLICATION` · `JOB` · `SYSTEM` · `ANNOUNCEMENT` — 타입별 아이콘·라벨·색상 |
| `title` | 제목 |
| `body` | 본문 (목록에서 최대 2줄 `line-clamp`) |
| `payload.href` | (선택) 있으면 **바로가기** 버튼으로 클라이언트 라우팅 |
| `createdAt` | ISO 문자열 — **상대 시각** 표시 (방금 전, N분 전, N시간 전, N일 전, 월·일) |
| `readAt` | `null`이면 미읽음, 값이 있으면 읽음 처리됨 |

초기 데이터는 `MOCK_NOTIFICATIONS` 상수 배열이며, 화면 상태는 `useState`로만 관리합니다. **새로고침 시 목업이 다시 초기화**됩니다.

### 1.2 UI · 인터랙션

- **페이지 헤더**: 섹션 라벨(NOTIFICATION), 제목「알림」, 목업 안내 문구.
- **전체 / 미읽음 토글**: 미읽음 선택 시 `readAt === null`인 항목만 표시. 미읽음 탭에 미읽음 개수 `(N)` 표시.
- **모두 읽음**: 미읽음이 1개 이상일 때 활성화. 클릭 시 모든 항목에 `readAt`을 현재 시각으로 설정 (로컬만).
- **미확인 알림 배너**: 미읽음이 1개 이상이고 사용자가 닫지 않았을 때만 표시. 문구에 미읽음 개수 포함. **닫기(X)** 시 세션 동안만 숨김(상태 초기화 시 다시 표시 가능).
- **알림 카드 목록**: 최신순(`createdAt` 내림차순) 정렬.
  - 미읽음: 강조 테두리·그림자, `NEW` 뱃지, 제목 굵게.
  - 읽음: 비교적 옅은 스타일.
- **바로가기**: `payload.href`가 있을 때만 노출, `useNavigate`로 이동.
- **읽음 처리**: 미읽음 항목만 버튼 노출, 해당 `id`만 `readAt` 설정 (로컬만).
- **빈 상태**: `EmptyState` — 전체/미읽음에 따라 제목·설명 문구 분기.

### 1.3 아직 없는 것 (연동 시 보완 대상)

- 로딩·에러·재시도 UI
- 서버 기준 **페이지네이션 / 무한 스크롤**
- **API·WebSocket** 호출
- 읽음/모두 읽음 후 **서버 반영** 및 실패 시 롤백
- 로그인 직후 배너를 **글로벌(다른 페이지)** 과 공유하는 `unread-count` 연동
- 사이드바·탑바 **알림 배지**와의 동기화
- 배너 닫기 상태의 **영속 저장**(localStorage 등)

## 2. 백엔드와 연동할 때 권장 순서

아래는 **의존 관계가 적은 것부터** 쌓아 가는 순서입니다. 백엔드와 **API 스펙(경로·필드명·인증)** 을 먼저 짧게 문서로 고정하는 것을 전제로 합니다.

### 2.1 계약 정리 (프론트·백 공통)

1. 알림 **JSON 스키마** 확정: `id`, `type`, `title`, `body`, `payload`, `createdAt`, `readAt`(또는 `read`) 등.
2. **목록 API**: `GET /notifications` — 쿼리(`cursor`/`limit`, `unreadOnly` 등)와 응답 형식.
3. **미읽음 개수 API**: `GET /notifications/unread-count` (또는 목록 메타에 포함).
4. **단건 읽음**: `PATCH /notifications/:id/read` (또는 동등).
5. **일괄 읽음**: `POST /notifications/read-all` (기획에 있을 때).
6. (후순위) **WebSocket** 이벤트명·페이로드가 REST 단건과 동일한지 여부.

이 단계에서 프론트의 `MockNotificationItem`을 **API DTO 타입**으로 이름·필드만 맞춰 갈아끼울 준비를 합니다.

### 2.2 프론트: API 레이어 추가

7. `src/api/Notification.ts`(예시 경로)에 `getNotifications`, `getUnreadCount`, `markNotificationRead`, `markAllNotificationsRead` 등 **fetch/axios 래퍼** 작성.  
8. 기존 프로젝트의 **인증 헤더·baseURL** 패턴(`Comment.ts` 등)에 맞출 것.

### 2.3 목록 화면을 서버 데이터로 전환

9. `Notification.tsx`에서 `MOCK_NOTIFICATIONS` 초기값 대신 **`useEffect` 또는 React Query 등으로 목록 로드**.
10. 로딩 스피너 / 에러 메시지·재시도 버튼 추가.
11. 서버가 **cursor 페이지네이션**을 주면 스크롤 하단에서 다음 페이지 요청(기획서 2단계).

### 2.4 읽음 동작을 서버와 일치

12. **읽음 처리** 클릭 시: 낙관적 업데이트(선택) + API 호출 → 성공 시 상태 확정, 실패 시 이전 상태 복구 및 토스트.
13. **모두 읽음** 동일 패턴.
14. 정책에 따라 **바로가기 클릭 시 자동 읽음** 여부를 백엔드와 맞춤.

### 2.5 로그인 직후·전역 미읽음 표시

15. 로그인 성공 직후 `getUnreadCount` 호출 → **0보다 크면** 알림 배너·토스트·헤더 배지 등에 반영 (Zustand/Context/React Query 캐시 등 팀 표준에 맞게).
16. 알림 페이지의 상단 배너는 **전역 `unreadCount`** 와 연동하거나, “이미 알림 페이지에 있음”일 때는 생략하는 식으로 UX 정리.

### 2.6 WebSocket (실시간)

17. 로그인 후 **단일 WS 연결** 수립(인증 방식은 백엔드 스펙 따름).
18. `notification:new` 수신 시 목록 상단에 merge·`unreadCount` 증가.
19. (선택) `notification:read` 등으로 **다른 탭**과 동기화.

### 2.7 마무리·품질

20. 라우트 가드: 비로그인 시 알림 페이지 접근 시 **로그인 유도**(이미 `Layout`에서 일부 처리 중이면 정책 통일).
21. `payload`의 `href`를 백엔드가 내려주는 **상대 경로 규칙**과 맞추고, 없는 라우트는 404 방지.
22. E2E 또는 수동 시나리오: 로그인 → 목록 로드 → 읽음 → 재진입 시 서버와 UI 일치.
