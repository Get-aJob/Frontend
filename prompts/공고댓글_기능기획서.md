# 공고댓글 기능기획서

## 1. 목표

- 사용자가 공고 상세 페이지에서 댓글을 조회/작성/삭제할 수 있게 한다.
- 댓글 기능은 공고 탐색 흐름을 방해하지 않고 자연스럽게 동작해야 한다.
- 초기 버전은 단순하고 안정적으로 구현하고, 확장은 2차에서 진행한다.

---

## 2. 범위

### 포함 (MVP)

- 공고 상세 진입
- 댓글 목록 조회
- 댓글 작성
- 본인 댓글 삭제
- 로딩/에러/빈 상태 UI

### 제외 (후순위)

- 대댓글(스레드)
- 댓글 좋아요/신고
- 수정 기능
- 멘션/알림 연동

---

## 3. 화면/라우팅 설계

- 전체 공고 페이지: `/posting-feed` (또는 기존 `/posting` 재사용)
- 공고 상세 페이지: `/posting/:jobId`
- 상세 페이지 하단에 댓글 섹션 배치
  - 댓글 입력 폼
  - 댓글 리스트
  - 페이지네이션 또는 무한스크롤(초기엔 페이지네이션 권장)

---

## 4. 데이터 설계(프론트 관점)

### Comment 타입(예시)

- id: string
- jobId: string
- authorId: string
- authorName: string
- content: string
- createdAt: string
- isMine: boolean

### 상태 관리

- 최소 단위:
  - comments
  - isLoading
  - isSubmitting
  - error
- 선택지:
  - 단기: 페이지 내부 `useState`
  - 확장 대비: `zustand` 스토어 분리 (`useCommentStore`)

---

## 5. API 계약(초안)

- GET `/jobs/:jobId/comments`
  - 댓글 목록 조회
- POST `/jobs/:jobId/comments`
  - 댓글 작성
- DELETE `/jobs/:jobId/comments/:commentId`
  - 댓글 삭제

공통 규칙:

- 인증 필요 (쿠키 기반 세션/토큰)
- 실패 시 사용자 메시지 일관화

## 8. 우선순위

### P0 (필수)

1. 상세 페이지 라우트/기본 UI 생성
2. 댓글 목록 조회 연결
3. 댓글 작성 연결
4. 본인 댓글 삭제 연결
5. 예외 상태(로딩/에러/빈 목록) 처리

### P1 (중요)

1. 댓글 입력 유효성(공백/길이 제한)
2. 낙관적 업데이트 또는 재조회 전략 정리
3. 접근성(키보드 포커스, 버튼 라벨)

### P2 (개선)

1. 댓글 페이지네이션/무한스크롤 개선
2. 작성 시간 포맷팅
3. UX polish (스켈레톤, 토스트)

## 10. 성공 기준(Definition of Done)

- 공고 상세에서 댓글 목록이 정상 조회된다.
- 로그인 사용자는 댓글 작성/삭제가 가능하다.
- 비로그인 사용자는 작성 시 인증 유도가 된다.
- 주요 예외 케이스에서도 화면이 깨지지 않는다.
- 코드가 기존 공고 도메인 구조와 일관된다.
